# **Steel Physical Products: Detailed Product Specifications**

## **1. STEEL CARD (NFC-Embedded Metal Card)**

### **Product Overview**
The Steel Card is a premium, NFC-enabled metal card that serves as a physical authentication token for the Steel membership ecosystem. Unlike traditional payment cards, it contains no visible sensitive data (no numbers, chip, or magnetic stripe) and uses encrypted dynamic tokens for secure authentication.

### **Physical Specifications**
```
DIMENSIONS:
- Standard ID-1 format: 85.6mm √ó 54.0mm √ó 0.8mm
- Material: 304 Stainless Steel with brushed finish
- Weight: 18-22 grams (heavier than plastic cards)
- Thickness: 0.8mm (standard card thickness for wallet compatibility)

FINISHES:
- Primary: Brushed gunmetal gray with laser-etched Steel logo
- Optional: Matte black PVD coating
- Edge: Beveled, polished edges for premium feel
- Surface: Anti-fingerprint coating

VISIBLE ELEMENTS:
- Front: Steel logo (minimalist, laser-etched)
- Back: Member name (optional, laser-etched)
- No card numbers, expiration dates, or CVV
- No magnetic stripe
- No EMV chip contacts
```

### **Technical Specifications**
```
NFC SYSTEM:
- Chip: NXP NTAG 424 DNA (or similar secure element)
- Frequency: 13.56 MHz (ISO/IEC 14443 Type A)
- Memory: 888 bytes (for UID, tokens, limited user data)
- Read Range: 2-4 cm (optimized for intentional tap)
- Encryption: AES-128/256 for dynamic token generation
- Standards: NFC Forum Type 4 Tag compliant

SECURITY FEATURES:
- Secure element with cryptographic functions
- One-time token generation capability
- Tamper-resistant design
- Water-resistant (IP67 equivalent)
- Temperature tolerance: -20¬∞C to 60¬∞C

COMMUNICATION:
- UID (Unique Identifier): 7-byte ISO standard
- NDEF records for contact sharing
- Custom URL scheme for deep linking to Steel app
- Fallback web URL for non-app users
```

### **Manufacturing Requirements**
```
PRODUCTION PROCESS:
1. Laser cut stainless steel blank
2. CNC mill slot for NFC inlay
3. Insert NFC inlay with secure adhesive
4. Seal with protective coating
5. Laser etch branding/member details
6. Quality control: NFC functionality, structural integrity

MATERIALS:
- Card body: 304 stainless steel (0.8mm)
- NFC inlay: PET-based with copper antenna
- Protective coating: Clear epoxy resin
- Packaging: Recycled cardboard with magnetic closure

CERTIFICATIONS NEEDED:
- NFC Forum certification
- CE/RoHS compliance
- ISO/IEC 7810 (ID-1 card standards)
- Payment card industry material standards (optional)
```

### **Packaging**
```
PACKAGING COMPONENTS:
- Outer sleeve: Matte black with debossed Steel logo
- Inner box: Magnetic closure, velvet lining
- Contents:
  - Steel Card
  - Microfiber cleaning cloth
  - Activation QR card
  - Membership guide (minimalist)
  - NFC compatibility card (explaining how to tap)
```

## **3. COMMON TECHNICAL INFRASTRUCTURE**

### **NFC Data Structure**
```
RECORD 1: NDEF Record
- Type: "text/x-vcard" for basic contact info
- Content: Name, basic contact (privacy-controlled)
- Trigger: Opens Steel app if installed

RECORD 2: NDEF Record  
- Type: URI record
- Content: https://steel.byexo.com/connect/[encrypted-ID]
- Fallback: Web view for non-members

RECORD 3: Custom Record
- Type: Application-specific
- Content: Encrypted session token
- Usage: Payment/authentication at partners
```

### **Security Protocol**
```
TOKEN GENERATION:
- Each tap generates new encrypted token
- Token valid for 30 seconds
- Token includes: Member ID + timestamp + session key
- Backend validates and maps to account

PAIRING SYSTEM:
- Core module/card paired to member's phone via app
- Bluetooth LE used for pairing (bracelet V2)
- Lost token can be disabled remotely
- New token requires re-pairing
```

DELETE ENTIRELY for MVP (Phase 1-2):
The Bracelet (Phase 2 ‚Üí Phase 3+):

Why: It's a second, radically more complex hardware product with significant R&D, certification, and supply chain challenges. It dilutes focus from the core card experience.

Interchangeable Band System:

Why: A "band ecosystem" is a startup trap. It's a fashion business, not a tech business, at this stage.

Action: If a bracelet happens later, launch with one band type (Sport Silicone). Add others only after proven demand.

"Guest Pass" System for Non-Members:

Why: Overcomplicates the social sharing flow. The goal is to grow the member network, not create a parallel non-member system.

Action: Sharing should be member-to-member only initially. Simpler.

Separate "Digital Only" ($199) and "Physical" ($495) Tiers:

Why: Creates a two-class system within the community and complicates marketing. The physical token is the magic.

Action: Have one membership tier that includes the digital platform. The physical card is a one-time purchase (e.g., $150-$250) or a clear, included perk of an "Elite" tier. Avoid fragmenting the network.

DEFER to Phase 2 or Later:
The Social "Tap-to-Connect" Layer:

Why: It's a secondary feature that depends on network density. The primary MVP job is Access & Payment.

Action: Launch social features after achieving critical mass in 1-2 cities. Don't build for a network that doesn't exist yet.

Why: It's a visionary community feature with no immediate ROI. It can be built after proving the core model.

API/Developer Platform:

Why: Premature. Need to solidify the core user experience and security model before opening it up.

SIMPLIFY Radically:
Payment Infrastructure:

Problem: "Closed-loop system (gift card regulations)" is a regulatory and UX minefield (pre-loading, limits, abandonment).

Simplification: Use a card-linking model like Apple Pay. Partner terminal authenticates via Steel, but the payment is routed via a user's linked Visa/Mastercard through a payment processor (Stripe/Braintree). This is faster to market, more familiar to users, and avoids being a "money transmitter." You can still take a lower transaction fee.
*Add On by User:* "Orchestration Layer" is what we are going for, for security and payment. This simplified version seems well but has it's cons as well. Needs to be thought through.

Steel Security Protocol:

Problem: "Dynamic cryptographic handshake per transaction" with a passive card may not be feasible.

Simplification: Use a secure, static token on the card that is immediately invalidated if the card is reported lost via the app. This is still far more secure than a printed credit card number. Phone pairing can be for loss protection, not per-transaction 2FA.

Simplification: Partner-facing iOS/Android App as the terminal. Employees use a company phone/tablet to tap member cards. Zero hardware cost for partners. Roll out dedicated hardware only for top-volume partners later.
